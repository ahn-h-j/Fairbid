# PR 리뷰 분석 작성 가이드

## 문서 구조

```markdown
# PR #{번호} 리뷰 분석

> **PR**: {PR 제목}
> **URL**: {PR URL}
> **리뷰어**: {리뷰어 목록 (예: CodeRabbit, GitHub Copilot)}
> **분석일**: {YYYY-MM-DD}

---

## {리뷰어명} 리뷰

### 1. {리뷰 제목}
- **파일**: `{파일경로:라인번호}`
- **내용**: {리뷰어가 지적한 내용 요약}
- **판정**: {✅ 수용 | ⚠️ 선택적 | ❌ 거부} {(Nitpick) - 사소한 제안인 경우}
- **AI 분석**: {AI가 분석한 기술적 배경 및 영향도}
- **결정**:
- **의견**:

---

## 요약

| 판정 | 개수 | 항목 |
|------|------|------|
| ✅ 수용 | 0개 | - |
| ⚠️ 선택적 | 0개 | - |
| ❌ 거부 | 0개 | - |
```

---

## 필드 설명

### AI가 작성하는 필드
| 필드 | 설명 |
|------|------|
| 파일 | 리뷰 대상 파일 경로와 라인 번호 |
| 내용 | 리뷰어가 지적한 내용 요약 |
| 판정 | AI가 리뷰 내용을 분석하여 판단한 중요도 (수용/선택적/거부) |
| AI 분석 | 기술적 배경, 영향 범위, 대안 제시 |

### 사용자가 작성하는 필드
| 필드 | 설명 |
|------|------|
| 결정 | 수용 / 거부 / 보류 |
| 의견 | 결정 사유 |

---

## 판정 기준

- **✅ 수용**: 리뷰어 제안이 명확히 맞고 반영해야 함
- **⚠️ 선택적**: 상황에 따라 다름. 프로젝트 맥락에 맞게 판단 필요
- **❌ 거부**: 리뷰어 제안이 현재 프로젝트에 맞지 않음

---

## Nitpick 표기 기준

- 코드 동작에 영향 없는 스타일/가독성 제안
- 성능 최적화 제안 (현재 문제없는 경우)
- 향후 확장성 관련 제안

---

## 파일 네이밍

```
pr-{번호}-{간단한설명}-review.md
```

예시:
- `pr-11-winning-bid-review.md`
- `pr-15-frontend-review.md`
